# 07 开发约束与验收

## Codex 开发规则

后续每次开发任务都必须遵守：

1. 先阅读 `AGENTS.md` 和 `docs/` 下所有规范。
2. 保持 Mock 模式可玩。
3. 不引入不必要依赖。
4. 不在前端暴露隐藏真相。
5. 不把最终判案交给 LLM。
6. 不临场修改案件真相。
7. 不把项目做成普通聊天应用。
8. 不把项目做成普通视觉小说。
9. 保持 TypeScript 类型尽量严格。
10. 修改脚本、目录、架构时同步更新 README。

---

## TypeScript 规范

要求：

- 尽量避免 `any`。
- 优先使用 `/shared/types.ts` 中的共享类型。
- API 请求和响应应有明确类型。
- 缺失 ID 或非法输入要安全处理。
- 前后端边界要清晰。

---

## 前端开发规范

要求：

- Phaser 负责主要游戏画面。
- 文本输入框可以用 HTML DOM Overlay。
- DOM 输入框在 Scene 销毁时必须清理。
- UI 文字要保证中文可读。
- 请求失败不能白屏。
- 玩家输入为空不能发送。
- 证据不能重复显示。
- 重要反馈要明显，例如：
  - 新证据发现
  - 矛盾命中
  - 嫌疑人状态变化
  - 结案评分

---

## 后端开发规范

要求：

- 后端是案件状态权威来源。
- Case Bible 只能后端持有。
- 结案前不能返回 Truth。
- 证据发现必须写入状态。
- 嫌疑人状态必须可更新。
- 最终评分必须规则化。
- Mock LLM 不需要 API Key。

---

## LLM 接入规范

MVP 不接真实 LLM。

后续接入真实 LLM 时必须遵守：

1. API Key 只能放在后端环境变量。
2. 前端不能直接请求 LLM。
3. 案件真相必须先结构化冻结。
4. NPC 只能看到自己的知识边界。
5. LLM 不能决定最终真相。
6. LLM 不能临场新增关键证据。
7. LLM 输出必须经过后端规则过滤。

---

## 手动验收流程

至少完成以下流程：

1. `npm install`
2. `npm run dev`
3. 打开前端页面
4. 进入标题界面
5. 点击开始案件
6. 查看案件简报
7. 调查后台休息室
8. 发现茶杯或毒物相关证据
9. 调查监控室
10. 发现前厅监控录像
11. 审问沈乔
12. 追问 21:12 离开前厅的矛盾
13. 命中矛盾反馈
14. 打开证据板
15. 查看已发现证据
16. 整理推理链
17. 提交最终报告
18. 获得评分和反馈
19. 重新开始案件

---

## 推荐测试点

如果添加测试，优先测试服务层：

- startCase 能创建案件
- getCase 不暴露隐藏真相
- investigate 能发现证据
- investigate 不重复发现证据
- interrogate 能返回 NPC 回复
- interrogate 能命中沈乔矛盾
- final-report 能正确评分
- final-report 提交后才能显示真相摘要

---

## MVP 完成定义

MVP 完成标准：

1. 新玩家可以从标题界面开始游戏。
2. 玩家可以调查 5 个地点。
3. 玩家可以审问 4 名嫌疑人。
4. 玩家可以发现重要证据。
5. 玩家至少可以命中 1 个核心矛盾。
6. 证据板可以展示已发现证据。
7. 玩家可以提交最终报告。
8. 结果界面可以展示评分和反馈。
9. 游戏无需真实 LLM API 也能完整游玩。
10. 结案前隐藏真相不会暴露给前端。
11. `npm run dev` 可以正常启动。
12. `npm run typecheck` 尽量保持通过。
